
# Chapter 5 暂存区（stage）概念 #
## 	5.1暂存区（stage）：git系统设计的一个重要概念 ##
执行`git status -s`时，会看到文件状态前出现不同位置不同颜色（在MSYSGIT下）的M标记

* 位于第一列的M（绿色）：版本库与暂存区的文件相比有改动  
* 位于第二列的M（红色）：工作区与暂存区 的文件 相比有改动

`git diff`  

* 不带参数的git diff：工作区与暂存区之间的差异  
* git diff HEAD：工作区与版本库之间的差异  
* git diff --cached：暂存区和版本库之间的差异  


## 5.2关于暂存区的理解 ##
 ![暂存区示例](http://i.imgur.com/pOcaJay.jpg)

暂存区是虚拟的“工作区”，是一个包含文件索引的目录树

执行git add命令时，暂存区的目录树被更新，工作区中的修改被写入到对象库的一个新的对象中，而该对象的ID将被记录到暂存区的文件索引中，但此时版本库并未受到影响。（工作区->暂存区）

当执行git commit命令时，暂存区的目录树会写入到版本库中，master分支会作相应更新。（暂存区->版本库）

`git reset HEAD`，暂存区的目录树被替换为master分支指向的目录树，工作区不受影响。（版本库->暂存区）

`git checkout .` . 暂存区->工作区，工作区中未添加到暂存区的修改会丢失。

`git checkout HEAD .` 版本库->暂存区->工作区，工作区和暂存区中未提交的改动会丢失。

## 8.3 Git checkout ##
`git checkout [<commit>] <path>`从暂存区检出，用暂存区覆盖工作区，不会改变HEAD头指针,用于覆盖工作区中的文件。<path>可以为`.`
`git checkout [<branch>]`（不使用路径），会改变HEAD头指针。一般用于**切换分支**。

`git checkout <branch>`更新HEAD指针指向branch分支，用branch分支指向的树更新暂存区和工作区

`git checkout [HEAD]`汇总显示工作区、暂存区和HEAD版之间的差异

`git checkout -- <filename>`（暂存区->工作区）用暂存区中的filename文件替换工作区中的filename文件， 当filename为“.”时，代表所有文件(当branch缺省时默认为暂存区)

`git checkout HEAD -- <filename>`（版本库->暂存区->工作区）用版本库中的filename文件替换工作区和暂存区中的filename文件， 当filename为“.”时，代表所有文件

`git checkout branch -- filename` 维持HEAD头指针不变，用branch中的filename文件替换暂存区和工作区中的filename文件，当filename为“.”时，代表所有文件


`git ckeckout [<commit>] -- <path>`形式从某一个提交中提取path所指向的文件以替换现有工作区中的文件，commit参数默认指向暂存区

`git ckeckout <commit>` 形式切换到某一个分支上，如果commit参数为HEAD~2形式则切换到detached branch未命名的分支

----

`git reset [<commit>]`更改分支游标的位置

`git reset`没有单独指向某一个文件的操作

默认情况是以版本库重置暂存区的内容，加上--soft参数则不会更改暂存区内容，加上--hard参数则会连工作区内容一起重置。commit默认指向HEAD指针的位置

如果是想撤销提交则一般使用`git reset HEAD`这样的形式

----
`git ckeckout` 和 `git reset`都可以用于撤销更改，两者的区别在于ckeckout并不更改分支游标的位置

`git ckeckout [<commit>] -- <path>`形式从某一个提交中提取path所指向的文件以替换现有暂存区和工作区中的文件，commit参数默认指向暂存区。不改变分支游标的位置，也不改变HEAD指针，仅仅是文件替换。

`git ckeckout <commit>` 形式切换到某一个分支上，如果commit参数为HEAD~2形式则切换到detached branch未命名的分支，暂存区和工作区当然都会随之改变（一般来说执行前后暂存区和工作区的改动都应该为空）。同样不会改变现有分支的游标，但是HEAD指针会移动到新的分支上。

`git reset`更改分支游标的位置，同时当然也会更改HEAD指针

`git reset`没有单独指向某一个文件的操作


默认情况是以版本库重置暂存区的内容，加上–soft参数则不会更改暂存区内容，加上–hard参数则会连工作区内容一起重置。commit默认指向HEAD指针的位置

如果是想撤销提交则一般使用`git reset HEAD~1`这样的形式，将现有分支游标切换到HEAD~1的提交上（连同HEAD指针一起），同时重置缓存区内容，相当于本地刚刚做了更改尚未git add。如果加了–soft则相当于本地做了更改并已经git add，如果加了–hard则相当于本地尚未做更改（本地更改全部丢失）。

#Chapter 9 Git stash:保存和恢复进度
##9.2 git stash的基本功能

`git stash`:保存暂存区和工作区的状态

`git stash list`：显示进度列表

`git stash pop`:恢复最新保存的工作进度，并将恢复的工作进度从工作进度列表中删除

`git stash apply`: 恢复最新保存的工作进度，不会将恢复的工作进度从工作进度列表中删除

`git stash drop [<stash>]`删除一个存储的进度，默认最新的进度

`git stash clear`:清除所有存储的进度

#Chapter 10 Git常见操作
##10.2关于文件删除

在工作区中rm删除文件不会影响到暂存区和版本库

类似于git add 将新增的文件加入暂存区，`git rm`会将在工作区中被删掉的文件从暂存区中删掉（工作区->暂存区），再次提交时，这些删除操作会被提交到版本库中

`git add -u` 将本地文件的所有更改（包括删除）全部记录到暂存区中

##10.3恢复删除的文件

回想git checkout 的用法，想恢复被删除的文件也很简单：  
`git checkout HEAD~1 -- filename`（ 版本库Repository->暂存区Stage->工作区working directory ），此时filename应该已经存在于暂存区和工作区中了。只需再次提交即可加入新的版本库。

##10.4文件重命名

与linux下的文件操作相似，git中的文件改名工作也是由文件移动命令git mv实现的。

`git mv oldname newname`一般来说在直接文件操作前加git都会将改动加入到暂存区。接下来只要提交就行了。

##10.5利用git tag给代码“打上烙印”

利用git tag命令可以给一个版本的代码生成一个“里程碑”标记当前和此后版本代码的版本号

#Chapter12 改变历史
##12.1修补式提交

修改提交信息。有时候提交了之后我们才会发现有一些零碎的工作没有完成，或者提交信息写的不合适，可是提交已经完成了又实在没有必要使用git checkout检出一遍。这时可以利用git commit --amend -m ""(如果只是单纯的改变提交信息而没有什么实际修改的话可能需要加上--allow-empty参数)

合并提交。如图所示如果最近的两次提交内容差不多的话，我们会想到能不能将其合并为一个提交，这件事的基本思路是：

1. 首先退回到两次提交之前：git reset --soft HEAD~2.这条重置命令让master分支回退到HEAD^^这个版本，但由于--soft参数，暂存区和工作区并没有受到任何影响。
2. 然后直接提交即可。git commit -m "coalescing."


## 12.2修改提交树
刚才上面的方法只能对于提交树的”末梢“修修改改，如果想要从中间将某个提交从历史上抹掉呢？  


![](http://i.imgur.com/MCLUBaj.png)

1. 如图，如果我们希望把提交D从提交树上抹掉的话，最直观的的思路是先会退到他前面的那个版本：`git checkout C`。注意，和刚才`git reset`不一样，后面不加特定文件名的`git checkout`都有”切换分支“的意思，也就是说我们现在不仅仅是将HEAD指针指向了C这个提交，更重要的是我们现在已经处在了**detached HEAD**状态。刚才`git reset`(将当前分支换一个位置)的时候，我们改变了master分支游标的位置，但是HEAD指针还是指向master分支的，如果你此时执行git branch的话，你会发现仍然处于master分支上。而`git checkout C`之后，master分支游标没变，可是HEAD指针变了，也就是说git现在不处在任何分支上，执行`git branch`会发现（no branch）。这种状态是比较危险的，它相当于一个尚未命名的分支.一般我们会让这种分支与其他分支合并或者直接用git checkout -b就地创建一个分支，给这个分支一个名分（插一句，觉得总是打git checkout太费事的同学可以直接`git config --global alias.ck "checkout"`给它起一个别名ck，以后直接`git ck`就行了）。
2. 下面我们使用cherry-pick命令，这个命令就是从众多的提交中挑一个应用在当前分支上。然后大家应该就知道了，就是把E和F加在C之上，那么D自然就被消掉了，可是有一点要注意，刚才那些提交的tag名是在master分支上命名的，现在我们回到了C，这个时候E和F都不存在于本分支中，因此用tag命名是找不到的。那么怎么办呢？别忘了master游标还是指向F没有变，因此`git cherry-pick master^`  `git cherry-pick master`就相当于将E和F又加回来了。
3. 还没有完，我们现在还是处在一个没有名字的”野分支“上，这可不行。因此还得回来git checkout master，可是这不就等于白做了吗？
4. 下一步最最关键：`git reset --hard HEAD@{1}` 首先git reset就是要改变master游标，也就是直接对本分支开刀。HEAD@{1}这种写法是第一次见到，以前我们写过HEAD~1，那相当于”跟踪“master所指向的那个提交链，找到现在提交的前一个提交。而HEAD@{1}相当于”跟踪“HEAD指针本身的行迹，指的是HEAD指针在移到现在这个分支前的指向，这条命令的意识就是，master游标移到了刚才那个”野分支“上，而原来的master分支倒是不存在了。如果用qgit等图形化工具查看的话大概就是这个样子的，原来的master分支”断掉了“。我们完成了一次”嫁接手术“。
